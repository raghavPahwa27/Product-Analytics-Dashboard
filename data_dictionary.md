# Data Dictionary — `customer_features`

One row = one unique customer (`customer_unique_id`).

Generated by `feature_engineering.py` from the Olist Brazilian E-Commerce dataset.

---

## Identity

| Feature | Type | Description |
|---|---|---|
| `customer_unique_id` | string | Olist's anonymous, persistent customer identifier. Survives across multiple orders — use this, never `customer_id`. |
| `state` | string | Brazilian state abbreviation (e.g. SP, RJ, MG). Derived from the customers table. |
| `city` | string | Customer's city. First value across all orders for that customer. |

---

## Volume & Spend

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `num_orders` | int | Total number of delivered orders placed. | Order frequency — a core RFM dimension. |
| `total_spend` | float (R$) | Sum of all item prices + freight across all orders. | Customer lifetime value (CLV). |
| `avg_order_value` | float (R$) | `total_spend / num_orders`. | Revenue efficiency per transaction. High AOV customers are worth retaining. |
| `total_products` | int | Total items purchased (not distinct — a qty of 3 counts as 3). | Purchase volume indicator. |
| `avg_basket_size` | float | `total_products / num_orders`. | How many items per order on average. Larger baskets = higher engagement. |
| `distinct_categories` | int | Number of unique product categories purchased. | Customer breadth — wide explorers vs. niche loyalists. |
| `distinct_sellers` | int | Number of unique sellers purchased from. | Marketplace breadth. High = trusts the platform, not a specific seller. |

---

## Timing & Recency

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `days_since_first_purchase` | int | Days from first purchase to the dataset's reference date. | Customer tenure. Long-tenured customers who stop are high churn risk. |
| `days_since_last_purchase` | int | Days from last purchase to the reference date. | **Recency** — the most predictive churn signal. The churn label is derived directly from this. |
| `purchase_frequency` | float | Orders per 30 days: `num_orders / max(tenure_days, 1) * 30`. | Buying cadence. Very low frequency is a churn precursor. |

---

## Delivery

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `avg_delivery_days` | float | Mean days between purchase timestamp and actual delivery. | Delivery experience. Slow delivery correlates with negative reviews and churn. |
| `pct_delayed` | float (0–1) | Fraction of orders delivered after the estimated date. | 1.0 = every order was late. High delay rate = dissatisfied customers likely to churn. |
| `avg_freight_ratio` | float (0–1) | Total freight cost / total spend. | Shipping cost sensitivity. High ratio may indicate the customer is paying disproportionately for shipping. |

---

## Satisfaction

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `avg_review_score` | float (1–5) | Mean review score across all orders with a review. NaN if customer left no reviews. | Satisfaction signal. Score ≤ 2 is a strong churn predictor. |

---

## Payment Behaviour

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `preferred_payment_method` | string | Payment type used most often (credit_card, boleto, voucher, debit_card). | Payment preference. Boleto users tend to be lower-income and may have different churn patterns. |

---

## Purchase Behaviour

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `weekend_purchase_ratio` | float (0–1) | Fraction of orders placed on Saturday or Sunday. | Behavioural segment signal. Weekend-heavy shoppers may differ in demographics and intent. |

---

## Target Variable

| Feature | Type | Description | Business Meaning |
|---|---|---|---|
| `churn` | int (0 / 1) | **1** if `days_since_last_purchase > 180`, else **0**. | The binary churn label used to train the prediction model in Part 4. |

---

## Churn Label — Design Decision

```
churn = 1   if days_since_last_purchase > 180
churn = 0   if days_since_last_purchase ≤ 180
```

**Reference date** = maximum `order_purchase_timestamp` in the dataset (≈ August 2018).

**Why 180 days?**
- Brazilian e-commerce repurchase cycles average 3–6 months.
- 180 days balances sensitivity (catching real churners) and specificity (not labelling seasonal shoppers as churned).
- It is a standard industry threshold for non-subscription e-commerce.
- Avoids data leakage: uses only the information observable within the dataset window.

**Known limitation:** customers who purchased in the final 180 days of the dataset window are labelled active even if they never returned. This is unavoidable without post-dataset-cutoff data and is standard practice in cohort-based churn modelling.
